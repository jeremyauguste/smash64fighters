❯ Thanks. Now can you help me understand how user's are assigned profile        
pictures? I want to know about how that data is stored across sessions.         

● Sure. Here's the full flow:

  At sign-up

  When a user creates an account, they pick an avatar from the icon grid in
  AuthModal. That choice gets saved into Supabase's built-in auth.users table
  under the user_metadata JSON column:

  supabase.auth.signUp({
    email,
    password,
    options: { data: { username, avatar } }
  })

  user_metadata is Supabase's native mechanism for storing arbitrary per-user
  data alongside the auth record. It persists in Supabase's database
  indefinitely — it's not a cookie or localStorage entry.

  ---
  Across sessions

  When the app loads, AuthContext calls supabase.auth.getSession() to check if a
   valid session token exists (stored in localStorage by the Supabase client
  automatically). If the session is valid, Supabase returns the full user object
   including user_metadata, which contains the avatar path. This is why the
  avatar survives page refreshes and browser restarts — the source of truth is
  the Supabase server, not the browser.

  // AuthContext.js
  supabase.auth.onAuthStateChange((_event, session) => {
    setUser(session?.user ?? null);
  });

  The user object flowing through your app always carries
  user.user_metadata.avatar.

  ---
  When a user changes their avatar

  On the Profile page, selecting a new icon calls supabase.auth.updateUser(),
  which writes the new avatar path back into user_metadata on the server:

  await supabase.auth.updateUser({
    data: { username: newUsername, avatar: selectedAvatar }
  });

  This also triggers onAuthStateChange, so AuthContext picks up the updated user
   object automatically — the auth bar and profile header re-render with the new
   avatar without a page reload.

  ---
  The profiles table

  There's a second copy of the avatar stored in your public profiles table
  (synced via upsertProfile() whenever the user saves changes). This exists
  solely so that other users' public profile pages can be fetched without
  exposing auth.users, which Supabase locks down by default. The profiles table
  is the one read when viewing someone else's /profile/:userId.

  ---
  In comments

  When a comment is posted, the avatar URL is snapshotted into the comments
  table row at that moment:

  avatar: user.user_metadata?.avatar ?? '/charIcons/smashball.png'

  This means if a user later changes their avatar, old comments still show the
  avatar they had at the time of posting — the comments table doesn't stay in
  sync with user_metadata.
